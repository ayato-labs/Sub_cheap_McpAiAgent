# ADR-0006: Rejection of Internal QA Loops (Semgrep)

- **Date**: 2026-06-15
- **Status**: Accepted
- **Deciders**: ayato-labs (User), Gemini CLI (Agent)

## Context
To improve the quality of code generated by inexpensive sub-LLMs, we considered integrating a local Quality Assurance (QA) loop within the MCP server using Semgrep. The proposed workflow was: Generate Code -> Run Semgrep -> If errors exist, send errors back to the sub-LLM to fix -> Return final code to Main AI. To save development costs, it was suggested to cap this loop at an arbitrary number (e.g., 3 retries).

## Decision
**Reject** the general-purpose integration of Semgrep and automated "generate-and-fix" retry loops within the sub-agent as a core feature. **However**, allow for "fail-fast" lightweight syntax validation (as defined in ADR-0019) with a strictly limited single-pass correction, serving as an exceptional rescue mechanism rather than a generalized QA loop.

## Rationale
1. **Violation of Core Philosophy**: The foundational philosophy of this project (defined in Conceptual Requirements 3.1) is that the sub-agent is a "stateless drafting factory." It must not perform complex autonomous reasoning, judgment, or self-correction (Agentic Workflows). QA, evaluation, and iteration are strictly the responsibilities of the Main AI or the CI/CD pipeline.
2. **Exceptional Rescue, Not General QA**: The "1-pass correction" allowed in ADR-0019 is restricted to immediate syntax failure cases. This differs from a full QA loop (which would perform static analysis, semantic checks, and comprehensive testing). It is designed to mitigate the cost of trivial syntax errors that block functionality, not to guarantee code quality.
3. **Arbitrary Constraints**: The original suggestion to use a hardcoded "3-loop limit" highlighted that the full QA feature itself was not essential. The current "1-pass" rescue is a pragmatic, bounded optimization rather than a design shift towards agentic behavior.
4. **Latency and Complexity**: Keeping the limit at "1 pass" ensures the MCP server maintains predictable latency and minimal complexity, adhering to the "stateless function" goal.
5. **Separation of Concerns**: The sub-agent's job is generation based on provided context. The Main AI's job is review and orchestration. Mixing these roles degrades the predictability and speed of the system. This exception for syntax repair is a narrow boundary explicitly defined to avoid role creep.

## Consequences
### Positive
- The MCP server maintains its extreme simplicity, remaining a true "stateless function."
- Predictable and fast latency for the Main AI, as the sub-agent only ever makes a single generation pass per request.
- No heavy external dependencies (like Semgrep binaries) required for users to install.

### Negative / Risks
- The Main AI may occasionally receive code with syntax errors or security flaws, requiring the Main AI to spend its own tokens to review and issue a correction command. This is accepted as a necessary trade-off for architectural purity.
