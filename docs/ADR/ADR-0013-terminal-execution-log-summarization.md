# ADR-0013: ターミナル実行ログの要約機能の導入

- **日付**: 2026年6月20日
- **ステータス**: 承認済み
- **決定者**: ayato-labs、Geminiエージェント

## 背景
ビルドログ、テスト結果、サーバーログなどのターミナル出力は非常に量が多くなることがあり、メインAI（Architect）のコンテキストウィンドウを大量に消費します。これにより、メインAIの推論能力の低下や、トークン制限によるコンテキストの喪失を招くリスクがあります。

## 決定事項
低コストなサブLLMを利用してログを要約し、重要な情報（エラー原因、実行結果）のみをメインAIに返すツール `execute_and_summarize` を導入します。

1. **無制限のシェル実行**:
   - `subprocess.run(shell=True)` を使用し、メインAIが意図したコマンドをホスト環境でそのまま実行します。

2. **ショートカットロジック**:
   - 出力内容が300文字未満の場合、サブLLMを介さず生のログを直接返却し、レイテンシとコストを最適化します。

3. **サブLLMによる要約**:
   - 300文字以上のログについては、`DRAFTING_MODEL`（デフォルト: `models/gemma-4-31b-it`）に要約を委任します。
    - 要約プロンプトでは、エラー、警告、および主要な結果に厳格に焦点を当てるよう指示します。
    - 要約の再現性と正確性を高めるため、サンプリング温度（Temperature）を低く（0.0）設定します。


4. **タイムアウト保護**:
   - 唯一のシステム保護策として実行時間制限（デフォルト60秒）を設けます。
   - タイムアウト発生時は、そこまで収集できた部分的なログを抽出し、サブLLMで要約して返却します。

5. **「ノーガード」戦略（ADR-0018にて更新）**:
    - ADR-0018にて、破壊的コマンドの実行を防ぐための最低限のセキュリティガードレールが導入されました。


## 影響
### メリット
- **トークン消費の劇的な削減**: 数千行のスタックトレースやビルドログを、数百トークンの要約に圧縮できます。
- **コンテキストの維持**: メインAIがコンテキスト制限に達することを防ぎ、一貫した推論能力を維持できます。
- **低レイテンシ**: 短い出力に対するパススルー処理により、不要なLLM呼び出しを回避します。

### デメリット／リスク
- **要約による情報の欠落**: サブLLMが重要な詳細を省略したり、誤認（ハルシネーション）したりする可能性があります。
- **セキュリティリスク**: 破壊的なコマンドが実行される可能性がありますが、これは開発効率とシンプルさを優先した設計判断によるものです。
