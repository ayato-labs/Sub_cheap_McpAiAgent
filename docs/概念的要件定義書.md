# 概念的要件定義書：Sub-cheap-McpAiAgent

## 1. プロジェクトの基本理念
本プロジェクトは、メインAI（高価格・高性能なフロンティアモデル）のコンテキスト（トークン消費量）を劇的に節約し、開発コストを最小化することを唯一絶対の目的とする。
高性能LLMの役割を「高度な推論・全体指揮（建築家/システムアーキテクト）」に絞り、最もトークンを消費する「コードの初稿作成・泥臭いファイル書き換え（アルバイト/プログラマー）」を、MCP（Model Context Protocol）を介して安価なLLM/ローカルLLMへ完全に委譲（アウトソーシング）する。

この明確な役割分担において、**「サブLLMには『完璧に動くコード』ではなく『叩き台（ドラフト）』を作らせるのが正解」**とする。安価なLLMのミスによる手戻りはシステムとして許容し、作業のアウトソーシングツールとして割り切る。

### 1.1. 基本ワークフロー
本システムにおけるエージェント間の情報の流れは、以下の通り極めて単方向かつシンプルに保たれる。

1. **メインAI（Claude Code等）**
   * 「やってほしいこと（指示）」と「参考情報など必要なコンテキスト」を抽出し、サブエージェントへ入力する。
2. **サブエージェント（本MCPサーバー＋安価なLLM）**
   * 入力された情報のみを頼りにタスク（コードの下書き・書き換え）を実行する。
3. **サブエージェント → メインAI**
   * 実行結果（コードまたは完了ステータス）のみを返す。「このように考えました」といった推論の途中式や経過報告は一切返さない。

## 2. 採用する方針と実装対象（何を・何で・なぜやるか）

### 2.1. FastMCP（Python）による独自実装
*   **方針**: 既存の汎用マルチエージェントOSS（MIT/Apache等含む）は一切採用せず、PythonのFastMCPライブラリを用いて自作する。
*   **理由**: 既存OSSは多機能ゆえに不要なプロンプトや自律思考（思考プロセス）が走り、トークンとレイテンシを無駄に消費する。今回の「コード下書き」という尖った用途には、FastMCPによるドキュメント操作ロジックの直書きが最速かつ最も実用的であるため。

### 2.2. 行指定によるピンポイントな差分書き込み
*   **方針**: メインAIからは「対象ファイル、指示、開始行・終了行」を受け取り、指定された行の範囲だけを切り出してサブLLMに投げる。
*   **理由**: 1000行あるファイル全体の読み書きをメインAIと往復させるとトークンが破滅する。変更が必要な「数行〜数十行」だけをサブLLMに処理させ、FastMCP側でファイルを直接書き換えることで、メインAIの往復トークンをほぼゼロにする。

### 2.3. ファイルの自動構築（ディレクトリ作成含む）
*   **方針**: 指定されたファイルパスが存在しない場合、FastMCPの内部ロジックで親ディレクトリを含めて自動生成し、新規ファイルとして下書きを書き込む。
*   **理由**: メインAIに「まずフォルダを作って、次にファイルを作って…」とステップを踏ませる指示の往復を省き、1回のツール呼び出しで「0からのコード構築」を完結させるため。

### 2.4. バックエンドLLMの二面運用（Gemini Flash & Ollama）
*   **方針**: Google AI Studioの無料枠（Gemini Flash等）と、ローカルLLM（Ollama経由のGemma 4 31Bなど）の2つを切り替えられる、または排他利用できる接続機構にする。
*   **理由**: 
    *   ネット接続環境や一般的なタスクでは、無料枠が手厚く長文に強いGemini Flashでコストをゼロにする。
    *   機密コードの処理、またはAPIのRate Limit（リクエスト制限）に達した際のフォールバックとして、完全無料でセキュアなローカルのOllamaを回す。

## 3. 排除する方針と非実装対象（何を・何で・なぜやらないか）

### 3.1. サブAIによる複雑な自律推論・ツール利用の排除
*   **方針**: サブLLM（Gemini/Gemma）に対して、独自にMCPツールを叩かせたり、ターミナルコマンド（`ls` や `grep`）を実行させたり、他のファイルを勝手に探索させたりする機能（Agentic Workflow）は絶対に実装しない。
*   **理由**: 
    *   サブLLMは「与えられたコンテキストと指示に基づいて、コードを出力するだけの純粋でステートレスな関数（下書き代行）」として扱う。
    *   **「思考（判断）コンテキスト」の意図的な遮断**: 必要な前提情報（コードの前後関係など）はメインAIからサブエージェントへしっかりと引き渡す。しかし、サブエージェントが引き渡すのは「結果（コード）」だけであり、途中式などの経過報告や推論プロセスは一切不要である。調査レポートの下書きを依頼した相手から、「私がどう考えてこの文章構成にしたのか」という裏の思考プロセスを受け取る必要がないのと同じである。判断と評価はすべてメインAIの責務とする。
    *   これにより、小型モデル特有の不安定さを吸収するための複雑な「LLMハーネスエンジニアリング」（エラーリトライ、無限ループ監視、パース機構の構築）を完全に不要にし、システムの堅牢性と高速性を保つ。

### 3.2. 高度なGit Conflictマージ機能の排除
*   **方針**: FastMCP側で複雑なGitのコンフリクト解消や差分チェックロジックは作り込まない。指定行を愚直に上書きする。
*   **理由**: 下書きコードの最終確認や既存コードとの競合チェックは、フロントエンド（CursorのDiff画面やメインAIのレビュー）の機能に委ねるべきであり、バックエンドのMCPサーバーを肥大化させるべきではないため。

### 3.3. 無料APIにおける「完全自動での機密データマスキング」の排除
*   **方針**: プログラムコード側で「個人情報を自動検知して隠す」ような複雑なセキュリティフィルターは実装しない。
*   **理由**: フィルターを過信したことによる情報漏洩を防ぐため。「機密データはローカル（Ollama）で動かす」というルーティングの決定権と責任は、人間またはメインAIのプロンプト設計側に持たせる。

### 3.4. 汎用的な構造化出力（Pydantic等）の厳格すぎるバリデーション
*   **方針**: サブLLMの出力をPydantic等でガチガチにバリデーションしてエラーで落とすような実装はしない。システムプロンプトによる「コードのみの出力」の徹底と、単純な文字列クレンジング（トリミング）に留める。
*   **理由**: 下書き（プロトタイプ）としての柔軟性を重視するため。文法エラーの修正やリファクタリング（清書）は、後続のステップで高性能なメインAIに行わせる方が効率的であるため。

---

## 4. トレードオフ（容認するリスク）
*   **スピード vs 確実性**: サブLLMが生成したコードがたまに動かないリスク（インポート文の不足など）は容認する。これはあくまで「下書き」であり、最終的なコードの整合性はメインAI（Claude）が辻褄を合わせる前提とする。
*   **開発コスト vs 機能**: サーバー側のコードは常に1ファイル、または最小限の構成を維持する。多機能にするくらいなら、プロンプトの工夫で解決する。
